W-ISLAND POLICY · v1.0

AI · API 사용 수칙

회사에서 AI 도구를 안전하고 일관되게 사용하기 위한 사내 가이드라인입니다. 보안 사고와 클라이언트 신뢰 손상을 예방하고, 누구나 같은 기준으로 일할 수 있도록 합니다.

v1.0 2026-06-18 전 직원 · 외주 협업자
승인된 도구 입력 금지 데이터 API · 비밀 관리 코드 · GitHub 결과물 검수 사고 대응

개요

왜 이 문서가 필요한가

AI 도구의 사용량이 빠르게 늘면서, 무분별한 사용으로 인한 보안 사고 · 클라이언트 자료 유출 · 저작권 문제가 실제 위험이 되었습니다. 본 수칙은 W-Island 구성원이 AI · API를 사용할 때 지켜야 할 최소한의 기준을 정리합니다.

"하지 말 것"을 정하는 것이 목적이 아니라, 안전한 사용 범위 안에서 빠르게 일할 수 있도록 공통 기준을 만드는 것이 목적입니다.

기본 원칙

모든 세부 규칙은 이 4가지 원칙에서 출발합니다

1 회사 계정으로만

개인 ChatGPT · Claude 무료 계정으로 회사 일을 하지 않습니다. 학습 옵트아웃이 보장된 회사 플랜(Team / Enterprise)만 사용합니다.

2 의심스러우면 묻기

이 데이터를 AI에 입력해도 되는지 확신이 없다면, 입력 전에 담당자에게 확인합니다. 사후 보고보다 사전 확인이 항상 빠릅니다.

3 AI 결과물은 초안

AI가 만든 텍스트 · 이미지 · 코드는 항상 사람이 검수한 뒤 외부로 나갑니다. 클라이언트 작업물은 특히 엄격하게.

4 사용 흔적 남기기

어떤 도구로, 어떤 프롬프트로 만들었는지 추적 가능해야 합니다. 문제가 생겼을 때 원인을 빠르게 찾을 수 있도록.

승인된 도구 (Whitelist)

업무에 사용할 수 있는 도구 목록입니다. 새 도구 도입은 신청 후 검토를 거칩니다.

카테고리 승인 조건부 금지
챗봇 · LLM ChatGPT Team, Claude Pro · Team, Gemini Workspace — 무료 ChatGPT(개인), 출처 불명 챗봇 사이트
이미지 · 영상 Higgsfield, Freepik AI, Photoshop · Firefly Midjourney (회사 계정만) 출처 불명 생성기, 개인 계정 사용
코딩 Claude Code, Cursor, Copilot Business — 개인 Copilot으로 회사 코드 작업
디자인 Figma AI Canva AI —

3

Do

API · 비밀 관리

키 하나가 유출되면 청구서 · 데이터 양쪽 모두 위험합니다. 가장 엄격하게 관리합니다.

저장 · 공유

✓
.env 파일에 저장 · .gitignore 등록
소스 코드에 키를 직접 박지 않습니다. .env 가 GitHub에 올라가지 않도록 반드시 .gitignore 에 등록
✓
팀 공유는 1Password · Bitwarden 사용
카카오톡 · 슬랙 DM · 이메일로 키 전달 금지
✓
회사용 · 개인용 키 분리
회사 결제 카드로 발급한 키와 개인 키를 섞지 않습니다

발급 · 회수

✓
키 대장 (Notion) 운영
누가 · 어떤 키 · 어디에 사용 · 발급일 · 한도 — 한 표에서 관리
✓
월 사용 한도(Spending limit) 필수 설정
OpenAI · Anthropic · Google 모두 한도 설정 기능 제공. 무한 청구 방지의 마지노선
✓
권한 최소화
read-only 로 충분하면 write 권한 부여 X. 키 범위(scope)는 좁게
✓
퇴사 · 외주 종료 시 24시간 내 회수
담당자가 즉시 폐기 · 재발급. 키 대장에 회수 기록

코드 · GitHub

회사 코드를 AI에 붙여넣을 때, 그리고 저장소 운영 시 지킬 규칙입니다.

코드를 AI에 붙여넣을 때

  • 승인된 도구(Claude Code · Cursor · Copilot Business 등)에서만 작업
  • 붙여넣기 전에 체크 — API 키 · 비밀번호 · 고객 데이터 · NDA 코드 포함 여부
  • 외부 클라이언트 코드는 클라이언트의 AI 사용 허용 여부를 먼저 확인
  • 실험은 회사 격리 저장소에서, 운영 코드는 정식 PR로

GitHub 저장소 필수 설정

설정상태비고

AI 생성 코드 검수

의존성 검토

AI가 추천하는 패키지명이 실제로 존재하는지 확인. 가짜 패키지명을 노린 악성 패키지 침투(slopsquatting) 사례가 늘고 있습니다.

라이선스 확인

GPL 등 전염성 라이선스 코드를 그대로 가져다 쓰면 회사 코드도 공개해야 할 수 있습니다.

보안 취약점

SQL injection · XSS · 인증 우회 같은 기본 취약점은 AI도 자주 만듭니다. PR 리뷰에서 점검.

사람의 리뷰

AI가 작성한 코드도 머지 전 사람이 반드시 리뷰. 자동 머지 금지.

코딩 에이전트 (Claude Code · Cursor)

자동 실행 권한이 있어서 일반 챗봇보다 더 엄격하게 다룹니다.

×
운영(production) 환경 접근 금지
로컬 · 개발 환경에서만 실행. 운영 DB · 운영 서버 직접 접근 금지
×
자동 승인(YOLO) 모드 금지
명령마다 사람이 확인하는 모드 권장. 권한 확장 명령은 특히 직접 확인
×
위험 명령은 사람이 직접
rm -rf · DB drop · force push · 키 회수 · 사용자 삭제 등
×
출처 불명 MCP 서버 연결 금지
시스템 권한을 가져갈 수 있음. 출처 · 코드 · 권한 범위 확인 후 추가
×
.env · 비밀 파일 폴더에서 실행 금지
에이전트가 의도치 않게 파일을 읽거나 외부에 노출시킬 위험
권장 작업 패턴에이전트에 작업을 맡길 때는 ① 작업 폴더를 격리 ② 변경 사항을 git diff 로 확인 ③ 테스트 통과 후 PR ④ 사람이 머지 — 4단계를 지켜주세요.

결과물 검수

AI 결과물이 외부로 나가기 전 확인할 사항입니다.

이미지 · 영상

  • wbL 콘텐츠 실사 원칙 유지 — 모델 · 제품 이미지는 AI 생성 금지, 실사 사용
  • 상표 · 인물 · 작품 침해 여부 확인
  • 손가락 · 텍스트 · 로고 왜곡 등 AI 흔적 점검
  • 출처(어떤 툴, 어떤 프롬프트) 기록

텍스트 · 카피

  • 사실관계 확인 (가격 · 일정 · 수치)
  • 브랜드 톤 일치 (비브아 / wbL 별도 톤)
  • 외국어 카피는 원어민 또는 별도 검수

코드

  • PR 리뷰 후 머지
  • 테스트 통과 확인
  • 의존성 · 라이선스 점검

클라이언트 작업물

  • 클라이언트의 AI 사용 동의 사전 확인
  • 납품 시 AI 사용 범위 고지 (계약에 따름)
  • 최종 결과물은 사람이 책임 검수

AI 사용 표기

외부 발행물에 AI 사용을 표기할지 여부는 브랜드 · 채널별로 다릅니다. 별도 가이드(브랜드별)를 따릅니다.

사고 대응

키 노출 · 데이터 유출 의심 시 행동 순서입니다.

1
즉시 보고
담당자(지정 필요)에게 슬랙 · 전화로 즉시 알림. "확실하지 않아서"라는 이유로 늦추지 않습니다
2
노출 자산 차단
노출된 키 즉시 폐기 · 재발급. 관련 세션 · 토큰 무효화
3
유출 범위 파악
어디까지 공유됐는지 · 누가 봤는지 · 학습에 사용될 가능성 점검
4
클라이언트 관련 시 고지
클라이언트 자료가 포함되면 빠르게 사실 고지. 늦은 고지가 더 큰 신뢰 손상
5
재발 방지책 문서화
사고 원인 · 대응 · 재발 방지책을 본 수칙에 반영. 같은 사고를 반복하지 않습니다

위반 시 조치

처벌이 목적이 아니라 재발 방지가 목적입니다.

2차반복 위반책임자 면담 + 사용 권한 일부 제한
