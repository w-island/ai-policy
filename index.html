W-ISLAND POLICY · v1.0

AI · API 사용 수칙

회사에서 AI 도구를 안전하고 일관되게 사용하기 위한 사내 가이드라인입니다. 보안 사고와 클라이언트 신뢰 손상을 예방하고, 누구나 같은 기준으로 일할 수 있도록 합니다.

v1.0 2026-06-18 전 직원 · 외주 협업자
승인된 도구 입력 금지 데이터 API · 비밀 관리 코드 · GitHub 결과물 검수 사고 대응

개요

왜 이 문서가 필요한가

AI 도구의 사용량이 빠르게 늘면서, 무분별한 사용으로 인한 보안 사고 · 클라이언트 자료 유출 · 저작권 문제가 실제 위험이 되었습니다. 본 수칙은 W-Island 구성원이 AI · API를 사용할 때 지켜야 할 최소한의 기준을 정리합니다.

"하지 말 것"을 정하는 것이 목적이 아니라, 안전한 사용 범위 안에서 빠르게 일할 수 있도록 공통 기준을 만드는 것이 목적입니다.

기본 원칙

모든 세부 규칙은 이 4가지 원칙에서 출발합니다

1 회사 계정으로만

개인 ChatGPT · Claude 무료 계정으로 회사 일을 하지 않습니다. 학습 옵트아웃이 보장된 회사 플랜(Team / Enterprise)만 사용합니다.

2 의심스러우면 묻기

이 데이터를 AI에 입력해도 되는지 확신이 없다면, 입력 전에 담당자에게 확인합니다. 사후 보고보다 사전 확인이 항상 빠릅니다.

3 AI 결과물은 초안

AI가 만든 텍스트 · 이미지 · 코드는 항상 사람이 검수한 뒤 외부로 나갑니다. 클라이언트 작업물은 특히 엄격하게.

4 사용 흔적 남기기

어떤 도구로, 어떤 프롬프트로 만들었는지 추적 가능해야 합니다. 문제가 생겼을 때 원인을 빠르게 찾을 수 있도록.

API · 비밀 관리

키 하나가 유출되면 청구서 · 데이터 양쪽 모두 위험합니다. 가장 엄격하게 관리합니다.

권한 등급 (Role-based Access)

키 발급 · 사용 · 콘솔 접근 권한을 4단계로 분리합니다. 등급이 높을수록 책임도 큽니다.

등급 대상 키 발급 키 사용 콘솔 접근
L3 · 관리자 대표 · 정책 책임자 ○ 최종 승인 ○ 전체
L2 · 팀장급 각 팀 리더 · 시니어 신청 · L3 승인 후 ○ 팀 범위
L1 · 일반 전 직원 (디자이너 · 기획자 등) × 발급된 키만 사용 ×
L0 · 외부 외주 · 인턴 · 단기 계약자 × 제한 키 · 만료일 필수 ×

키 신규 발급은 L2 팀장급 이상만 신청할 수 있으며, 최종 승인은 L3 관리자가 합니다. L1 일반 직원은 이미 발급된 키를 받아서 사용만 합니다.

키 발급 절차

1
신청서 제출 (L1 · L2)
사용 목적 · 예상 사용량 · 사용 기간 · 환경(prod / dev) · 권한 범위(scope) 명시. Notion 양식 사용
2
L2 팀장 1차 검토
기술적 필요성 · 권한 범위 · 대안(기존 키 활용) 검토
3
L3 관리자 최종 승인
비용 한도 · 보안 영향 확인 후 발급. 운영 키는 반드시 L3가 직접 발급
4
키 대장 등록
발급자 · 사용자 · 키 종류 · 환경 · 한도 · 발급일 · 만료일을 Notion 키 대장에 기록

환경별 키 분리

운영 · 개발 · 테스트 환경의 키를 절대 섞지 않습니다. 하나의 키가 모든 환경을 커버하지 않도록 합니다.

환경용도한도 (참고)접근 가능자
Production운영 서비스 (ai-hub, 자사몰 등 실제 사용자 노출)실 사용량 기반 · 알림 필수L2 이상 + 운영 담당자
Staging배포 전 검증 환경월 $50 ~ $100L1 이상
Development개발 · 디버깅 · 학습 · 실험월 $20 ~ $50L1 이상
운영 키는 로컬 PC에서 사용 금지개발 · 디버깅 · 실험은 반드시 development 키로. 운영 키는 운영 서버의 환경변수에만 존재해야 합니다.

계정 · 콘솔 보안

✓
API 콘솔 2단계 인증(2FA) 필수
OpenAI · Anthropic · Google Cloud · GitHub 모두 2FA ON. SMS보다 인증 앱(Authy · 1Password · Google Authenticator) 권장
✓
결제 계정은 회사 명의로 1~2개만
개인 명의 계정으로 회사 키 발급 금지. 퇴사 시 계정 인수가 불가능하고 결제 정보가 개인에게 묶이는 문제 발생
✓
제공자별 Organization · Project 단위로 분리
OpenAI Organization, Anthropic Workspace, Google Cloud Project를 프로젝트별로 분리해서 사용량 · 권한을 격리
✓
콘솔 직접 접근은 L2 이상
L1 일반 직원은 발급된 키만 사용. 콘솔에서 키 발급 · 한도 변경 · 멤버 초대 권한은 팀장급 이상
✓
IP 제한 · 호출 출처 제한 적용
제공자가 지원하는 경우 회사 · 운영 서버 IP만 허용. 모바일 앱이라면 도메인 · 패키지명 제한 활용

저장 · 공유

✓
.env 파일에 저장 · .gitignore 등록
소스 코드에 키를 직접 박지 않습니다. .env 가 GitHub에 올라가지 않도록 반드시 .gitignore 에 등록
✓
팀 공유는 1Password · Bitwarden 만 사용
카카오톡 · 슬랙 DM · 이메일 · Notion 본문에 키 평문 전달 금지. 비밀 관리 도구만 사용
✓
권한 최소화 (Least Privilege)
read-only 로 충분하면 write 권한 X. 키 scope · 호출 가능 모델 · 호출 가능 메서드를 가능한 좁게 설정
✓
운영 키는 비밀 관리 서비스에 보관
중요 운영 키는 1Password Vault · AWS Secrets Manager · GitHub Encrypted Secrets 등 전용 저장소 사용

모니터링 · 한도

사용 한도 (Spending Limit)

  • 모든 키에 월 사용 한도 필수 설정
  • 예상 사용량의 1.5배 정도가 적정
  • 한도 도달 시 자동 차단되도록 hard limit 설정

알림 임계값

  • 한도의 50% · 80% · 100%에 자동 알림
  • Slack · 이메일로 L2 · L3에게 전달
  • 평소 대비 3배 이상 급증 시 즉시 알림

주간 리뷰 (L2 담당)

  • 매주 월요일 사용량 리포트 확인
  • 예상 외 호출 패턴 · 비활성 키 점검
  • 이상 발견 시 L3에 즉시 보고

감사 로그 (Audit Log)

  • 키 발급 · 회수 · 권한 변경 이력 영구 보존
  • 분기 1회 L3가 전체 키 · 사용 패턴 검토
  • 이상 사용 발견 시 즉시 조사 · 폐기

정기 로테이션 (Key Rotation)

키는 주기적으로 폐기 · 재발급합니다. 오래된 키일수록 노출 가능성이 누적됩니다.

키 종류주기비고
운영 키 (Production)3개월분기마다 재발급 · 무중단 전환 (이전 키 1주 병행)
개발 · 테스트 키6개월반기마다 재발급
외부 협업자 키 (L0)프로젝트 종료 시 즉시최대 3개월 만료 강제
의심 상황 발생 시즉시주기와 무관하게 폐기 · 재발급

회수 · 폐기

✓
퇴사 · 외주 종료 시 24시간 내 회수
L3 관리자가 즉시 폐기 · 재발급. 키 대장에 회수 기록 · 회수자 서명
✓
프로젝트 종료 시 즉시 회수
관련 키는 모두 폐기. 같은 키를 다른 프로젝트에 재사용 금지
✓
노출 의심 시 "묻지 마" 폐기
확실하지 않아도 의심되면 즉시 폐기. 늦은 폐기가 가장 큰 비용

.gitignore 예시

# .gitignore 에 반드시 포함
.env
.env.local
.env.*.local
*.pem
*.key
credentials.json
secrets/
config/secrets.*
키 노출이 의심되면즉시 해당 키를 폐기 · 재발급하고 L3 담당자에게 보고합니다. GitHub에 올라간 키는 1분 안에 자동 봇이 훔쳐가서 청구서가 폭발할 수 있습니다.

제공자별 권한 설정 가이드

L0~L3 등급을 각 AI 도구에서 실제로 어떻게 구현하는지 정리했습니다.

Anthropic (Claude)

Claude는 ① API Console(개발용)과 ② Claude.ai · Claude Code(채팅 · 코딩 도구)의 권한 분리 방식이 다릅니다.

① API Console console.anthropic.com

3겹 구조: Organization → Workspaces → Roles

  • Organization — 회사 단위. 결제 카드 1개, 전체 Spend limit
  • Workspaces — 프로젝트별 분리. 각 워크스페이스마다 별도 키 · 별도 한도 · 별도 멤버
  • Roles — Primary Owner / Admin / Billing / Developer / Member
W-Island 등급Anthropic Role비고
L3 관리자Organization Primary Owner결제 · 멤버 · 한도 · 모든 워크스페이스 관리
L2 팀장급Organization Admin 또는 담당 Workspace Admin키 발급 · 팀원 초대 · 사용량 확인
L1 일반특정 Workspace의 Developer그 워크스페이스 키만 사용. 다른 프로젝트 접근 X
L0 외부멤버로 초대 X만료 API 키만 전달. 프로젝트 종료 시 키 폐기

W-Island 추천 Workspace 구성

# W-Island Organization 내부
ai-hub-prod      # 운영 · 실제 사용자 노출
ai-hub-dev       # 개발 · 디버깅 · 실험
vivoir           # 비브아 프로젝트
wbl              # wbL 프로젝트
design-team      # 디자인 팀 일반 사용

② Claude.ai · Claude Code (구독)

API 키 대신 Claude Team 플랜으로 멤버 관리. 학습 옵트아웃이 기본 적용됩니다.

  • Admin — 멤버 초대 · 청구 · 사용량 (L3)
  • Member — 채팅 · Projects 사용 (L1 · L2)
  • 외주(L0)에게는 일회성 게스트 접근 금지 — 차라리 회사 PC에서 작업하도록

③ Claude Code CLI 추가 권한 제어

사용자 PC에서 동작하는 도구라 .claude/settings.json으로 추가 제어가 가능합니다.

{
  // 외주(L0)에게 환경 줄 때 권장 설정
  "permissions": {
    "allow": ["Read", "Grep", "Glob"],
    "deny": [
      "Bash(rm:*)",
      "Bash(git push:*)",
      "Bash(git reset --hard:*)",
      "Write(.env*)"
    ],
    "ask": ["Edit", "Write", "Bash"]
  }
}

OpenAI (ChatGPT · API)

구조는 Anthropic과 유사: Organization → Projects → Members → API Keys

W-Island 등급OpenAI Role비고
L3 관리자Organization Owner결제 · 멤버 · 모든 Project 관리
L2 팀장급Project Owner해당 Project 내 키 발급 · 멤버 관리
L1 일반Project Member특정 Project 키 사용
L0 외부멤버 초대 X · 키만 전달Project 단위 격리 + 만료일

Settings → Limits에서 Project별 월 한도 · 모델별 한도 · 알림 임계값 설정 가능. ChatGPT Team/Enterprise는 별도 워크스페이스에서 멤버 관리.

Google (Gemini · Cloud)

Google Cloud의 IAM (Identity and Access Management)으로 권한 분리. Project 단위로 격리.

W-Island 등급Google Cloud IAM Role비고
L3 관리자Owner · Billing AdministratorProject 생성 · 결제 · 모든 권한
L2 팀장급Editor + API Keys Admin키 발급 · 리소스 관리

AI 생성 코드 검수

의존성 검토

AI가 추천하는 패키지명이 실제로 존재하는지 확인. 가짜 패키지명을 노린 악성 패키지 침투(slopsquatting) 사례가 늘고 있습니다.

라이선스 확인

GPL 등 전염성 라이선스 코드를 그대로 가져다 쓰면 회사 코드도 공개해야 할 수 있습니다.

보안 취약점

SQL injection · XSS · 인증 우회 같은 기본 취약점은 AI도 자주 만듭니다. PR 리뷰에서 점검.

사람의 리뷰

AI가 작성한 코드도 머지 전 사람이 반드시 리뷰. 자동 머지 금지.

코딩 에이전트 (Claude Code · Cursor)

자동 실행 권한이 있어서 일반 챗봇보다 더 엄격하게 다룹니다.

×
운영(production) 환경 접근 금지
로컬 · 개발 환경에서만 실행. 운영 DB · 운영 서버 직접 접근 금지
×
자동 승인(YOLO) 모드 금지
명령마다 사람이 확인하는 모드 권장. 권한 확장 명령은 특히 직접 확인
×
위험 명령은 사람이 직접
rm -rf · DB drop · force push · 키 회수 · 사용자 삭제 등
×
출처 불명 MCP 서버 연결 금지
시스템 권한을 가져갈 수 있음. 출처 · 코드 · 권한 범위 확인 후 추가
×
.env · 비밀 파일 폴더에서 실행 금지
에이전트가 의도치 않게 파일을 읽거나 외부에 노출시킬 위험
권장 작업 패턴에이전트에 작업을 맡길 때는 ① 작업 폴더를 격리 ② 변경 사항을 git diff 로 확인 ③ 테스트 통과 후 PR ④ 사람이 머지 — 4단계를 지켜주세요.
